# 複数ユーザー対応時の懸念点

今後ユーザーを増やす際に検討・対応が必要な項目を整理したものです。

---

## 1. データのユーザー分離

### 1.1 現状ユーザー単位で分離されているもの
| データ | 備考 |
|--------|------|
| **UserConfig**（目標・パーソナル） | `userId` で分離済み |
| **MealLog / MealItem** | `userId` で分離済み（/api/meals は JWT 必須） |

### 1.2 現状「全員で1共有」のもの（要検討）

| データ | 懸念 |
|--------|------|
| **DailyData** | あすけん・Strong・歩数が「日付」で1件のみ。複数ユーザーになると「誰のデータか」が分からず、全員が同じ内容を見ることになる。ユーザーごとに取るなら `userId`（または `ownerId`）の追加が必要。 |
| **AiEvaluation** | AI評価がユーザー紐づきなし。複数ユーザーで「誰の評価か」が分からず、履歴が混ざる。`userId` の追加と、取得・保存時のフィルタが必要。 |
| **SyncLog** | id=1 の1レコードのみ。「誰の同期結果か」が分からない。ユーザーごとにログを持つか、あるいは「default ユーザー用」と割り切るか方針が必要。 |

---

## 2. 認証・フロント

| 懸念 | 内容 |
|------|------|
| **ログインUIがない** | 現状、どの画面もログインしていない。パーソナル・履歴・ダッシュボードはすべて「認証なし = default」で動いている。 |
| **API にトークンを送っていない** | `/api/config`, `/api/days`, `/api/day/*`, `/api/sync`, `/api/ai/*` などは `Authorization` ヘッダーを付けずに fetch している。JWT を渡す実装が必要。 |
| **ユーザー切り替えがない** | 複数ユーザーになっても「いま誰として見ているか」を選ぶ手段がない。ログイン/ログアウトと、ログイン中のユーザーを API に渡す必要がある。 |
| **getUserIdForConfig のフォールバック** | 認証なしのとき常に `"default"` を返している。ログイン必須にすると、未ログイン時は 401 にするか、引き続き "default" のままにするか方針が必要。 |

---

## 3. 同期（あすけん・Strong）の「誰のデータか」

| 懸念 | 内容 |
|------|------|
| **あすけん認証が1セット** | `ASKEN_EMAIL` / `ASKEN_PASSWORD` が環境変数で1つのみ。複数ユーザーがそれぞれ自分のあすけんを紐づけるには「ユーザーごとの認証情報」の保存が必要（DB または暗号化したストア）。 |
| **セッションが1ファイル** | `secrets/asken-state.json` が1つ。複数ユーザーなら `asken-state-{userId}.json` などユーザーごとのセッション管理が必要。 |
| **Strong が1パス/1アカウント** | `STRONG_DATA_PATH` や Google Drive（`GOOGLE_*`, `GOOGLE_DRIVE_STRONG_FOLDER_ID`）が1つ。ユーザーごとにフォルダやアカウントを切り替える設計が必要。 |
| **同期 API にユーザーが渡っていない** | `POST /api/sync` は「誰のデータを取るか」を受け取っていない。認証で userId を決め、そのユーザー用のあすけん/Strong 設定で同期する形に変える必要がある。 |
| **cron が「誰用」か未定義** | 朝5時の同期・AI評価が「default のみ」「全ユーザーを回す」のどちらか、仕様の決定と実装が必要。 |

---

## 4. API ごとの認証の差

| API | 現状 | 懸念 |
|-----|------|------|
| **/api/meals** | `getUserIdFromRequest` で JWT 必須（なしなら 401） | 多ユーザー時もこのまま「ログイン必須」にしやすい。 |
| **/api/config** | 認証なし可 → `getUserIdForConfig` で "default" | ログイン必須にするなら、ここも JWT 必須にし、sub を userId に使う必要がある。 |
| **/api/days** | 認証なし → "default" の目標でスコア計算 | 履歴一覧を「ログイン中のユーザーだけ」にするなら、userId で DailyData をフィルタする前提でスキーマ変更が必要。 |
| **/api/day/[date]** | 認証なし。DailyData を日付で1件取得 | ユーザー分離するなら、userId に紐づく DailyData のみ返す必要がある。 |
| **/api/sync** | 認証なし。環境変数のあすけん/Strong で1ユーザー分のみ取得 | 誰の同期かを受け取り、そのユーザー用の認証情報で実行する必要がある。 |
| **/api/ai/evaluate**, **/api/ai/history** | 認証なし。AiEvaluation に userId なし | 評価をユーザー単位にするなら、保存・取得ともに userId で分離する必要がある。 |

---

## 5. User と UserConfig の対応関係

| 懸念 | 内容 |
|------|------|
| **UserConfig.userId と User.id** | UserConfig の `userId` は JWT の `sub` や `"default"`。Supabase の場合、`sub` は UUID で、User テーブルの `id` と一致させる運用が一般的。MealLog は `User.id` を参照しているので、JWT の sub = User.id にしておくと、MealLog と UserConfig で同じユーザーを指せる。 |
| **"default" の扱い** | 未ログイン時用の "default" は、User テーブルに存在しない。ゲスト利用を続けるなら "default" のまま。ログイン必須にするなら、 "default" は廃止し、未認証は 401 にすることになる。 |

---

## 6. その他

| 懸念 | 内容 |
|------|------|
| **ダッシュボード・履歴の表示** | 表示しているカロリー/PFC/歩数/Strong はすべて DailyData の「1人分」。DailyData をユーザー分離しない限り、全員が同じ数値を見る。 |
| **AI 評価のプロンプト** | `generateDailyPrompt` は `getGoals()` で目標を読んでいる（現状は default）。ユーザーごとの目標で評価するには、評価実行時に userId を渡し、`getGoals(userId)` を使う必要がある。 |
| **パフォーマンス・コスト** | ユーザー数に応じた同期・AI評価の実行回数増加。cron で「全ユーザー分」を回す場合、レート制限や実行時間・コストの検討が必要。 |

---

## 対応の優先度イメージ

1. **認証フローとフロント**  
   ログインUIの追加、ログイン中の JWT を API に付与する。
2. **DailyData のユーザー分離**  
   `userId`（または `ownerId`）を追加し、取得・同期・表示をすべて「そのユーザー分」に限定する。
3. **AiEvaluation のユーザー分離**  
   `userId` を追加し、保存・取得をユーザー単位にする。
4. **同期のユーザー紐づけ**  
   ユーザーごとのあすけん/Strong 認証情報の保存と、同期 API・cron で「誰のデータを取るか」を決める。
5. **SyncLog**  
   ユーザーごとにするか、「default 用の1件」のままにするか方針を決めてから変更。

この順で検討・実装すると、多ユーザー対応を段階的に進められます。

## 7. あすけん認証情報の保存場所と「管理者から見えない」設計

ユーザーごとにあすけんのデータを取る場合、**ID・パスワードなどをどこに置くか**は次の2通りです。

### 7.1 ローカル（ブラウザ・端末のみ）に置く場合

| メリット | デメリット |
|----------|------------|
| サーバー側にパスワードが残らないので、DB漏洩や管理者が見る心配がほぼない。 | **cron で「朝5時に自動同期」ができない。** そのユーザーはログインして「今すぐ取得」を押したときだけ同期される。 |
| 実装が比較的シンプル（送信しないだけ）。 | 別デバイスからログインした場合は、またその端末で認証情報を入力する必要がある。 |

→ **管理者から絶対に見えない**ようにしたいならローカルは有効だが、「サーバー側の定期同期」はあきらめる形になる。

### 7.2 設定画面で入力し、サーバーに保存する場合

| メリット | デメリット |
|----------|------------|
| サーバー側で同期できるので、**cron による朝5時同期**などが可能。 | サーバー（DB）に載せる以上、**平文のまま保存すると管理者やDB漏洩で見える**。 |
| どのデバイスからログインしても同じ認証情報で同期できる。 | **「管理者から見えない」にするには、暗号化して保存する設計が必須。** |

→ **「設定画面で設定し、管理者からは見えない」にするなら、サーバーに保存するにしても「暗号化して保存」が前提になる。

### 7.3 推奨: 設定画面で設定し、暗号化してサーバーに保存する

- **保存する場所**  
  設定画面で入力したあすけんのメール・パスワード（および必要なら Strong / Google 系）は、**ユーザーごとのテーブル（例: UserSecret や UserCredential）に「暗号化した値」だけ**を保存する。
- **管理者から見えないようにする**  
  - **平文でDBに保存しない。** ログにも出さない。  
  - サーバー側で「復号して使う」ときだけメモリ上で復号し、用が終わったらすぐ破棄する。  
  - 暗号化鍵の設計が重要（下記）。

**暗号化の設計例**

| 方式 | 概要 | トレードオフ |
|------|------|----------------|
| **アプリ共通のマスター鍵で暗号化** | サーバーだけが持つ鍵（環境変数など）で暗号化。DBを直接見られても平文は分からない。 | 管理者がサーバー環境にログインできれば、その鍵で復号は理論上可能。「運用で管理者を信頼する」前提になる。 |
| **ユーザーごとの鍵（パスワード由来）** | ユーザーが「同期用マスターパスワード」を設定し、そこから導出した鍵で暗号化。復号はそのパスワードが分かる時だけ可能。 | 管理者はパスワードを知らない限り復号できない。一方で、cron で自動同期するには「ユーザーがログインしたときにだけ復号してメモリに載せる」「cron ではそのユーザー分はスキップ」など、運用設計が必要。 |
| **クライアント側で暗号化してから送る** | ブラウザで鍵を導出し、送信前に暗号化。サーバーは暗号文だけを保存し、復号鍵は持たない。 | サーバー・管理者は原理的に復号できない。ただしサーバー側で同期を実行するには、ユーザーがログインしているセッション時に「一時的に」復号した値を渡すか、別の仕組みが必要。 |

**結論**

- **「設定画面で設定し、管理者からは見えない」** にしたいなら、**少なくとも平文保存は避け、暗号化して保存する設計**が必須。
- さらに「管理者がサーバーに触れても復号できない」まで求めるなら、**クライアント側で暗号化してから送る**か、**復号鍵をユーザーだけが持つ方式**を検討する必要がある（その場合は cron 同期の仕様と両立させる設計が別途必要）。
- いずれにしても、**ローカルだけに置くか、設定画面＋暗号化保存か**のどちらにしても、「管理者から見えない」を満たすには、**平文をサーバーに載せない**ことが前提、という整理になる。

**方針**: cron 同期や他デバイスでの利用を考えると、**サーバー／DB に暗号化して保存する方式をベター**とする。暗号化鍵はアプリのマスター鍵（環境変数）でよい場合は実装が簡単。管理者のアクセスをより制限したい場合は、ユーザー由来の鍵やクライアント側暗号化を検討する。

## 8. 有料化・広告収益を見据えた機能の差別化

- **前提**: ユーザーごとに分けるなら、取得するデータ（DailyData・AiEvaluation・同期結果など）も**すべてユーザー単位で別**にするのが当然。1ユーザー＝1単位のデータとして扱う。

からだノートで有料プランや広告で収益を得る場合、**無料では得られない価値**をはっきりさせると、課金・広告許容の理由になりやすい。差別化の候補を整理する。

### 8.0 有料化するなら「他アプリからのインプット」を前提にする

**現状の課題**  
あすけん・Strong だけをサポートしているため、**それ以外のアプリを使っている人には刺さらない**。食事記録は MyFitnessPal / Lifesum / あすけん など、トレーニングは Strong / Fitbod / ストラバ など、人によってツールが違う。

**有料化する場合の方向性**  
有料化して収益を得るなら、**他アプリでのインプットも含めて対応・調整できる**ことが前提になる。

- **食事・栄養**: あすけん以外（MyFitnessPal、Lifesum、Apple Health 連携、手動入力、CSV インポートなど）からの取り込みを用意する。
- **トレーニング**: Strong 以外（他アプリの API、手動入力、CSV、ウェアラブル連携など）からの取り込みを用意する。
- **歩数・活動**: あすけん経由以外（Apple Health / Google Fit / ウェアラブル API など）も選択肢にする。

そうすることで「自分が使っているアプリのデータをまとめて見たい・AI で評価したい」というニーズに応えられ、あすけん・Strong ユーザー以外にも刺さる。有料の価値は「**どのアプリで記録していても、からだノートに集約して分析・AI 評価できる**」というところに置く。

### 8.1 有料（プレミアム）で差別化しやすい機能

| 分野 | 機能例 | 差別化のポイント |
|------|--------|------------------|
| **AI・分析** | 週次レポートの自動生成・要約、月次トレンド、減量/増量フェーズ別のアドバイス | 無料は「今日の評価」のみ、有料は過去も含めた分析・レポート。 |
| **AI・分析** | システムプロンプトのカスタム（目的・制約を自分用に調整）、評価回数無制限 | 無料は1日1回 or 手動のみ、有料はcron含め無制限＋プロンプト編集。 |
| **データ** | 履歴の保持期間（無料は3〜6ヶ月、有料は無制限）、エクスポート（CSV/PDF） | 長く使う人・記録を残したい人には強い訴求。 |
| **同期** | 自動同期の頻度（無料は手動のみ、有料は朝5時などcron）、**複数連携**（あすけん・Strong に加え他アプリからの取り込み） | 手間を減らす価値で課金理由にしやすい。**他アプリ対応は有料の前提**（8.0 参照）。 |
| **ユーザー** | 家族・チームで共有（複数ユーザー、または1アカウントで複数プロフィール） | 個人利用は無料、複数人で使うなら有料。 |
| **UX** | 広告非表示、テーマ・レイアウトのカスタム、優先サポート | 無料は広告あり、有料は広告なし＋快適さ。 |

### 8.2 広告で収益を取る場合の線引き

- **無料ユーザー**: ダッシュボード・履歴・パーソナル設定は使えるが、**バナーやインタースティシャル広告を表示**。AI評価は1日1回まで、履歴は直近3ヶ月のみ、など制限をかける。
- **有料ユーザー**: 広告非表示 ＋ 上記の「プレミアム機能」の一部または全部を提供。  
→ 「広告を見たくない」「もっと便利に使いたい」という理由で有料に誘導できる。

### 8.3 からだノートならではの差別化（刺さる訴求）

> **重要な修正**: 当初の訴求は「機能の差別化」に寄りすぎていた。実際のペルソナ（後述 Section 9）の動機は「登山ができる体をキープすること自体が私生活全体の意識・レベルを上げる」という生き方の軸であり、そこへの訴求に変更する。

| 訴求 | 具体例 |
|------|--------|
| **「動ける体でいること」が目的、数値はその手段** | ダイエットや筋肥大が目標ではなく「山に登れる、動ける体をキープする」が軸。スコアが「今日の自分は登山に向けてどうか」を毎日教えてくれるのは、この層にしか刺さらない唯一の訴求。 |
| **私生活全体の意識・レベル向上の可視化** | 「動ける体を維持しているとき、仕事も生活も意識が変わる」という感覚を数値とAIコメントで裏付けられる。単なるヘルスケアアプリではなく「生き方を支えるツール」というポジション。 |
| **あすけん＋Strong の一元化** | 食事（あすけん）と筋トレ（Strong）を1つのダッシュボードで見られる。有料で「PFCとトレーニング量の相関」「週次レポートで食事と筋トレのバランス」を可視化。 |
| **目的別の目標・プロンプト** | 無料は標準目標のみ、有料は「登山前の調整」「維持期」「増量」など目的別テンプレート＋AIプロンプトのカスタム。 |

### 8.4 料金モデルの例

- **無料**: 基本機能（今日のサマリー、履歴、手動同期、AI評価は手動1日1回）、広告あり、履歴は直近3ヶ月。
- **月額・年額**: 広告非表示、自動同期（cron）、AI評価無制限、履歴無制限、週次レポート、エクスポート、カスタムプロンプト。
- **家族プラン**: 上記に加え、複数ユーザー（または複数プロフィール）で同じアカウントを共有。

まずは **「広告非表示」「自動同期」「履歴保持期間」** の3つを有料の柱にすると、実装負荷と訴求のバランスが取りやすい。
そのうえで、有料化で幅広く刺させるなら **「他アプリからのインプット対応」**（食事・トレーニング・活動の取り込み元を増やす）を product の前提にしておく。

## 9. マーケティング戦略

> Section 8 は「機能と収益モデル」の整理だったが、**誰に・どうやって届けるか**が設計されていなかった。技術が完成しても人が来なければ収益ゼロ。ここではその観点を補足する。

---

### 9.1 ターゲットペルソナ（コアペルソナ）

```
年齢・属性  : 32歳・会社員（デスクワーク系 / IT・コンサル・営業など）
習慣        : 週2〜3回筋トレ、月1〜2回登山
使用ツール  : あすけん（食事記録）、Strong（トレーニング記録）

本質的な動機:
  「山に登れる、動ける体をキープすること」が目的であり、
  それを維持することで仕事・私生活全体の意識・レベルが上がると感じている。
  ダイエットでも筋肥大でもなく「使える体でいる生き方」を軸に
  食事とトレーニングを管理したい。

抱えている課題:
  - 食事（あすけん）と筋トレ（Strong）を横断して見るツールがない
  - 「今日の自分は登山・筋トレに向けてどうか」をAIに評価してほしい
  - バラバラなデータを一元管理し、週・月のトレンドを把握したい
  - 記録が習慣化しているので、分析・フィードバックに価値を感じる
```

**このペルソナが「刺さる」理由**: 競合アプリは「ダイエット」か「筋肥大」のどちらかに寄っており、「動ける体を軸にした私生活全体の意識向上」という軸に応えるツールは存在しない。

**登山コミュニティの年齢層と獲得戦略の前提**:

登山・トレッキングの実際のユーザー層は **30〜70代** に広く分布しており、中心は **40〜50代** 。
一方、からだノートのコアペルソナ（あすけん＋Strongを既に使っているデジタル記録習慣者）は **30〜40代前半** に集中する。

```
登山人口全体（YAMAP等のコミュニティ）
  30代 ---- 40代 ---- 50代 ---- 60代 ---- 70代
  │          │          │          │          │
  ↑          ↑
  コアペルソナが存在する層
  （あすけん・Strongを既に使っている）
  ここへのリーチが最優先

  ↑ -------- ↑ -------- ↑
  ブランド認知層（登山×栄養管理の啓発で将来的に取り込む）
```

→ **まず30〜40代の「記録習慣がある登山者」に刺す。40代以降は啓発コンテンツで時間をかけて引き寄せる。**

### 9.2 プロダクトポジショニング

```
競合の立ち位置:
  食事管理アプリ → カロリー・PFCを記録してダイエット
  筋トレアプリ   → ボリュームを記録して筋肉を増やす
  健康管理アプリ → 体重・歩数で「なんとなく健康」を維持する

からだノートの立ち位置（競合不在のポジション）:
  「動ける体でいること」を軸に食事×筋トレ×活動を一元管理し、
  AIが「今日の自分は山に登れるか？」を毎日評価してくれる。
  数字の管理が目的ではなく、「私生活全体の意識・レベルを上げる生き方」
  を数値とAIで支えるツール。
```

**1行ポジショニング（キャッチコピー案）:**

```
✗ 「食事とトレーニングを管理するアプリ」 → 競合多数、差別化なし
✓ 「動ける体でいることが、人生のクオリティを上げる。」
```

### 9.4 「日常設計軸」としてのコンテンツ戦略

からだノートの本質的な価値は「記録を振り返るツール」ではなく、**「山に登れる体を維持するために、今日をどう過ごすかの判断軸を与えるツール」** である。

この視点でコンテンツを作ると、アプリの宣伝ではなく**共感が先に生まれる**。

```
✗ 「からだノートというアプリを作りました」 → 宣伝にしか見えない
✓ 「月1登山のために週3筋トレと食事をこう設計している」 → 自分ごとに見える
```

**コンテンツのタネ（実体験ベース）:**

| テーマ | 内容 | 刺さる読者 |
|---|---|---|
| 登山前の調整 | 「登山2週前からたんぱく質を意識的に増やしている。AIスコアで確認」 | 登山×栄養に興味がある人 |
| 日常とスコアの関係 | 「残業続きでスコアが下がった。食事より睡眠が原因とAIに言われた」 | データで管理したい人 |
| あすけん×Strongの連携 | 「食事記録と筋トレ記録を繋げて初めて、筋トレ日は消費が増えていることが数字で見えた」 | どちらかを使っている人 |
| 体の調子と私生活の変化 | 「動ける体をキープしていると仕事の意識まで変わる、という話」 | 生き方の軸として共感する人 |
| 失敗談・正直な話 | 「先月の山行でバテた。振り返るとたんぱく質が足りなかった」 | リアリティを求める人 |

→ これらを**日記感覚で発信し続けることがアプリの宣伝より先に信頼を生む**。「この人が使っているから気になる」という順序。

### 9.5 口コミが起きる体験設計

口コミが起きる条件: **具体的な変化の体験 ＋ シェアしたくなる感情**

からだノートで設計すべき口コミポイント:

- 「AIに今日の食事とトレーニングを評価してもらったら、登山スコア〇〇点と言われた」
- 「先月より登山適性スコアが上がった。AIに○○が原因と言われて改善した」
- 「あすけん×Strongを繋げて初めて、食事とトレーニングの相関が見えた」

→ **無料でこの体験を味わわせることが最優先。** 課金前に「こんなことわかるのか」を体験させ、その感動がシェアの動機になる。広告や機能制限より体験の質を先に磨く。

### 9.6 有料化のタイミングと判断指標

**技術的な多ユーザー対応より先に、以下の指標を確認してから投資判断する。**

```
有料化に進む前の確認指標:
  □ 週1回以上アクティブなユーザーが10人以上いる
  □ 「これ友達に薦めたい」と言われた事例が3件以上ある
  □ 「自動同期が欲しい」「履歴を長く見たい」という要望が複数届いている

上記が揃ってから:
  1. 有料プランの設計・実装
  2. 多ユーザー対応の技術投資（Section 1〜7）
  3. 集客チャネルの拡大

揃う前に多ユーザー対応に入ると:
  → 技術コストは高いが需要が不明なままになるリスクがある
```
